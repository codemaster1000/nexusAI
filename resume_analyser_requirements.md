# Resume Analyser — Project Requirements

## 1. Project Overview

An AI-powered resume analysis tool that accepts a resume PDF and a job description, then produces a structured analysis including a match score, skill gap identification, section-by-section feedback, and on-demand rewriting of weak sections. Implements dynamic model routing via OpenRouter — lightweight models for extraction tasks, powerful models for complex analysis and rewriting.

---

## 2. Functional Requirements

### 2.1 Input Handling

**FR-01 — Resume Upload**
- System must accept resume in PDF format via drag-and-drop or click-to-upload
- Maximum file size: 5MB
- System must extract full text from resume using pdfplumber (Node.js: pdf-parse)
- System must display extracted text preview so user can confirm extraction was successful
- System must handle single and multi-page resumes
- System must display a clear error if PDF is image-based (scanned) with no extractable text. Skip images present alongside extractable text gracefully.

**FR-02 — Job Description Input**
- System must provide a textarea for user to paste job description
- Minimum 100 characters required — enforce with validation message
- Maximum 5000 characters accepted
- System must display character count in real time
- Job description must be preserved across multiple analyses in the same session

**FR-03 — Input Validation**
- Analyse button must be disabled until both resume and job description are provided
- System must display field-level error messages for missing inputs
- System must confirm both inputs are present before making any API call

---

### 2.2 Core Analysis Engine

**FR-04 — Match Score Calculation**
- System must produce an overall match score between 0 and 100
- Score must be generated by LLM based on semantic alignment — not keyword counting
- Score must be broken down into sub-scores:
  - Technical Skills Match (0-100)
  - Experience Relevance (0-100)
  - Education Fit (0-100)
  - Soft Skills Alignment (0-100)
- Overall score = weighted average: Technical 40% + Experience 30% + Education 20% + Soft Skills 10%
- Score must be displayed as animated circular progress indicator
- Score colour coding: 0-40 red, 41-70 amber, 71-100 green

**FR-05 — Skills Analysis**
System must identify and categorise skills into three groups:

- **Matched Skills** — skills present in both resume and job description
- **Missing Skills** — skills required by job description but absent from resume
- **Bonus Skills** — skills in resume not required by JD but potentially valuable

Each skill must be displayed as a colour-coded chip:
- Matched: green chip
- Missing: red chip
- Bonus: blue chip

Missing skills must be ranked by importance — how prominently they appear in the job description.

**FR-06 — Section-by-Section Feedback**
System must analyse each standard resume section and provide specific feedback:

- **Summary/Objective** — is it tailored to this role? Too generic?
- **Work Experience** — are achievements quantified? Action verbs used? Relevant experience highlighted?
- **Skills Section** — missing critical skills? Outdated technologies listed?
- **Education** — relevant coursework highlighted? Certifications present?
- **Projects** — do projects demonstrate required skills? Impact clearly stated?

Each section must receive:
- A status indicator: Strong / Needs Improvement / Missing
- 2-3 specific, actionable feedback points — not generic advice
- An option to request AI rewrite of that section

**FR-07 — ATS Compatibility Check**
- System must check resume for common ATS (Applicant Tracking System) issues:
  - Presence of tables or columns — ATS parsers struggle with these
  - Use of headers/footers — often missed by ATS
  - Non-standard section headings — e.g. "My Journey" instead of "Experience"
  - Missing contact information
  - Uncommon fonts or special characters
- Each issue flagged with severity: Critical / Warning / Suggestion
- ATS score displayed separately from match score

**FR-08 — Keyword Optimisation**
- System must extract top 10 keywords from job description by frequency and importance
- System must check which keywords are present and absent in resume
- Missing high-priority keywords displayed with suggested sentences to naturally incorporate them
- System must NOT suggest keyword stuffing — flag if keyword density becomes unnatural

---

### 2.3 AI Rewriting Feature

**FR-09 — Section Rewrite Request**
- User must be able to click "Rewrite This Section" on any section with "Needs Improvement" status
- System must send original section text + job description + specific feedback to LLM
- Rewrite must be targeted — improve the section specifically for this job description
- System must show original and rewritten versions side by side
- User must be able to copy rewritten text with one click

**FR-10 — Rewrite Quality Rules**
LLM prompt for rewriting must enforce:
- Preserve all factual information — never invent experience or skills
- Use strong action verbs — led, built, reduced, increased, implemented
- Quantify achievements where possible — add placeholder like [X%] if numbers unknown
- Match tone and keywords from job description
- Keep approximately same length as original — no padding

**FR-11 — Full Resume Rewrite**
- User must be able to request full resume rewrite after section analysis
- System must rewrite all "Needs Improvement" sections in sequence
- Display progress indicator — "Rewriting Experience section... 2/4"
- Final output displayed as complete rewritten resume text
- Copy all button to copy entire rewritten resume

---

### 2.4 Model Routing Logic

**FR-12 — Automatic Model Selection**
System must route to different models based on task type:

| Task | Model | Reason |
|---|---|---|
| Text extraction + basic parsing | mistralai/mistral-7b-instruct | Simple structured task, cheap |
| Skills identification and matching | anthropic/claude-haiku-4-5 | Moderate reasoning needed |
| Match score calculation + sub-scores | anthropic/claude-sonnet-4-5 | Nuanced judgment required |
| Section feedback generation | anthropic/claude-sonnet-4-5 | Detailed qualitative analysis |
| Section rewriting | anthropic/claude-sonnet-4-5 | Creative + accurate, high stakes |
| ATS compatibility check | mistralai/mistral-7b-instruct | Rule-based, simple extraction |
| Keyword optimisation | anthropic/claude-haiku-4-5 | Moderate NLP task |

**FR-13 — Model Transparency**
- Each analysis card must display which model was used
- Total token usage and estimated cost displayed in analysis summary
- Model routing decisions logged for debugging

### 2.5 Structured Output Requirements

**FR-14 — JSON Response Schema**
LLM must return analysis in strict JSON format. Backend must validate against this schema:

```json
{
  "overall_score": 78,
  "sub_scores": {
    "technical_skills": 82,
    "experience_relevance": 75,
    "education_fit": 80,
    "soft_skills": 70
  },
  "matched_skills": ["Python", "React", "Node.js", "REST APIs"],
  "missing_skills": ["Docker", "Kubernetes", "CI/CD"],
  "bonus_skills": ["Drone programming", "Embedded systems"],
  "sections": {
    "summary": {
      "status": "needs_improvement",
      "feedback": [
        "Too generic — does not mention the target role or company",
        "Missing quantifiable achievements",
        "No mention of key required skills: Docker, cloud platforms"
      ]
    },
    "experience": {
      "status": "strong",
      "feedback": [
        "Good use of action verbs",
        "Consider quantifying the user growth metric in Project X"
      ]
    }
  },
  "ats_issues": [
    {
      "issue": "Resume uses a two-column layout",
      "severity": "critical",
      "suggestion": "Switch to single-column format for ATS compatibility"
    }
  ],
  "top_keywords": {
    "present": ["Python", "API", "React"],
    "missing": ["Docker", "microservices", "agile"]
  }
}
```

**FR-15 — JSON Parsing Resilience**
- Backend must attempt JSON.parse on LLM response
- On parse failure: retry once with stricter prompt instructing JSON only output
- On second failure: return structured error to frontend with raw LLM response
- Never crash silently — always return something useful to user

---

### 2.6 Results Display

**FR-16 — Analysis Dashboard**
Results must be displayed in a structured dashboard with these sections:
- Overall match score — prominent, top of page, animated
- Sub-score breakdown — horizontal bar charts
- Skills analysis — three columns: matched, missing, bonus
- Section feedback — expandable accordion per section with rewrite button
- ATS compatibility — checklist style
- Keyword optimisation — table with present/missing columns
- Cost summary — tokens used, estimated cost, models used

**FR-17 — Export Results**
- User must be able to download full analysis as PDF report
- PDF must include all scores, feedback, and rewritten sections
- Export button prominently placed at top of results

**FR-18 — Re-analyse**
- User must be able to update job description and re-analyse same resume
- Re-analysis must not require re-uploading resume
- Previous analysis cleared and replaced with new results

## 3. Non-Functional Requirements

### 3.1 Performance
- Initial analysis (score + skills) must complete within 15 seconds
- Section feedback must stream progressively — not wait for all sections to complete
- Rewrite requests must begin streaming within 3 seconds

### 3.2 Cost Management
- Full analysis estimated cost must not exceed $0.05 per run using model routing
- Without routing (all claude-sonnet-4-5): approximately $0.15 per run
- Model routing saves approximately 65% cost — this is a key talking point
- Display cost breakdown so user understands value of routing

### 3.3 Privacy
- Resume content must never be stored permanently on server
- All analysis data cleared from memory after response sent
- No logging of resume text or personal information
- Clear privacy notice displayed before upload

### 3.4 Accuracy
- LLM must be explicitly instructed not to invent skills or experience
- Rewrite prompts must enforce factual accuracy preservation
- System must flag if confidence in extraction is low — e.g. scanned PDF

## 5. API Endpoints

### POST /analyse
- Accepts: multipart/form-data — resume PDF + job_description text
- Returns: JSON analysis object matching FR-14 schema
- Processing: extract → parallel analysis tasks → model routing → structured response

### POST /rewrite
- Accepts: { section_name, original_text, job_description, feedback }
- Returns: Server-Sent Events stream of rewritten section
- Processing: construct rewrite prompt → route to claude-sonnet-4-5 → stream

### POST /rewrite-full
- Accepts: { sections_to_rewrite[], job_description, original_sections{} }
- Returns: SSE stream with progress events + rewritten sections
- Processing: sequential section rewrites with progress updates

### POST /export
- Accepts: { analysis_data, rewritten_sections }
- Returns: PDF file download
- Processing: generate formatted PDF from analysis data

## 7. Prompt Engineering Specifications

### Analysis Prompt Structure

```
System: You are an expert resume analyst and hiring consultant with 10 years 
of experience. Analyse the resume against the job description provided.
Return ONLY a valid JSON object matching the exact schema below. 
No preamble, no explanation, no markdown code fences. Raw JSON only.

Schema: [insert full JSON schema from FR-14]

Rules:
- Base match scores on semantic relevance, not just keyword presence
- Missing skills must be ranked by how critical they are to the role
- Feedback must be specific and actionable — never generic
- Never invent or assume skills not explicitly stated in the resume
- ATS issues must reference specific problems found in the text

Resume:
[extracted resume text]

Job Description:
[job description text]
```

### Rewrite Prompt Structure

```
System: You are an expert resume writer. Rewrite the provided resume section 
to better match the target job description.

Rules — strictly enforce all:
1. Preserve ALL factual information — never invent experience, skills, or metrics
2. Use strong action verbs: led, built, reduced, increased, shipped, designed
3. Quantify achievements — if numbers are unknown use [X%] or [X users] as placeholders
4. Incorporate these missing keywords naturally: [keywords list]
5. Match approximately the same length as original
6. Do not pad with filler phrases

Original Section ([section_name]):
[original text]

Job Description:
[job description]

Specific Issues To Fix:
[feedback points from analysis]

Rewrite the section now:
```
